# 通信协议与状态管理

## 核心理念

三个角色（规划器、生成器、评估器）之间不直接对话。它们通过读写约定的文件进行通信，就像接力赛传递接力棒一样。

整个流程只有一个全局状态文件 `state.md`，记录当前进展和下一步该谁执行。

---

## 文件体系

### 文件清单与职责

| 文件 | 写入者 | 读取者 | 生命周期 |
|------|--------|--------|----------|
| `plan.md` | 规划器 | 生成器 + 评估器 | 规划完成后不变，除非需求变更 |
| `sprint.md` | 生成器 | 评估器 | 每轮迭代被覆盖 |
| `evaluation.md` | 评估器 | 生成器 | 每轮评估被覆盖 |
| `state.md` | 当前执行者 | 下一个执行者 | 持续更新 |

### 文件格式要求

- 使用 Markdown 格式
- 结构化、层次分明
- 不使用 emoji（保持专业性）
- 每个文件开头必须标注写入者和时间

```markdown
<!-- 写入者: {Planner|Generator|Evaluator} -->
<!-- 时间: {ISO 8601} -->
<!-- Sprint: {N} -->
```

## 完整生命周期

### 阶段 1：规划

```
触发: 人类给出产品需求
执行: 规划器
产出: plan.md
状态: PLANNING → PLAN_READY
```

1. 人类将需求传递给规划器
2. 规划器生成 `plan.md`
3. 规划器写入 `state.md`：
   ```
   status: PLAN_READY
   current_role: Generator
   sprint: 0
   ```
4. 规划器工作结束，控制权交给生成器

### 阶段 2：迭代循环

```
触发: state.md 中 current_role 为 Generator
执行: 生成器
产出: sprint.md + 可运行的软件
状态: ITERATING → SPRINT_READY
```

```
触发: state.md 中 current_role 为 Evaluator
执行: 评估器
产出: evaluation.md
状态: EVALUATING → ITERATING 或 DELIVERED
```

这个循环持续进行，直到评估器判定通过。

### 阶段 3：交付

```
触发: evaluation.md 中判定为 PASS
执行: 评估器（更新状态）
状态: DELIVERED
```

1. 评估器写入最终 `evaluation.md`
2. 评估器更新 `state.md`：
   ```
   status: DELIVERED
   current_role: null
   final_score: {分数}
   total_sprints: {N}
   ```
3. 流程终止

## state.md 格式

```markdown
<!-- 更新者: {当前角色} -->
<!-- 更新时间: {ISO 8601} -->

# 运行状态

## 当前状态
{PLANNING | PLAN_READY | ITERATING | SPRINT_READY | EVALUATING | DELIVERED | PIVOT}

## 当前执行者
{Planner | Generator | Evaluator | null}

## Sprint 信息
- 当前 Sprint: {N}
- 迭代轮次: {M}
- 总迭代次数: {K}

## 进度摘要
{用 2-3 句话概括当前进展}

## 功能完成度
| 优先级 | 总数 | 已完成 | 状态 |
|--------|------|--------|------|
| P0 | {N} | {N} | {全部完成/进行中/未开始} |
| P1 | {N} | {N} | {同上} |
| P2 | {N} | {N} | {同上} |

## 关键事件日志
{按时间倒序记录关键事件}

### Sprint {N} - 轮次 {M}
- [{时间}] {角色}: {事件描述}

### Sprint {N-1} - 轮次 {L}
- [{时间}] {角色}: {事件描述}

## 待决事项
{如果有需要人类介入的事项，记录在此。正常迭代中此节应为空。}
```

## 状态流转

```
PLANNING ──────────────────────► PLAN_READY
  (规划器工作)                     (plan.md 就绪)

PLAN_READY ────────────────────► ITERATING
  (生成器开始工作)

ITERATING ─────────────────────► SPRINT_READY
  (生成器完成本轮)

SPRINT_READY ──────────────────► EVALUATING
  (评估器开始工作)

EVALUATING ────────────────────► ITERATING    (未通过)
                                DELIVERED    (通过)
                                PIVOT        (需要重新规划)

DELIVERED ─────────────────────► (终止)

PIVOT ─────────────────────────► PLANNING    (退回规划器)
```

### PIVOT 条件

评估器在以下情况可以触发 PIVOT（退回规划器重新规划）：

1. 生成器连续 3 轮总分提升 < 5 分，且仍然 < 60 分
2. 生成器明确表示当前规划无法实现
3. 评估器判定当前实现方向与产品定义存在根本性偏差

PIVOT 发生时，评估器在 `evaluation.md` 中说明原因，并在 `state.md` 中设置 `status: PIVOT, current_role: Planner`。

## 异常处理

### 生成器遇到无法自行解决的问题

生成器在 `sprint.md` 的"自检结果"中记录，并标记为"待决事项"：

```markdown
## 自检结果
- 无法解决的问题：{描述}
- 已尝试：{尝试过什么}
- 建议：{需要什么帮助}
```

评估器在下一轮评估中判断：
- 如果该问题不影响核心功能 → 降低相关维度评分，继续迭代
- 如果该问题阻塞核心功能 → 考虑触发 PIVOT

### 评估器与生成器的分歧

如果生成器认为评估器的反馈不合理（例如要求实现 `plan.md` 中没有定义的功能），生成器在 `sprint.md` 中说明理由，但不停止工作。评估器在下一轮评估中重新审视。

**核心原则**：流程永远不应该因为分歧而停滞。在分歧未解决之前，评估器的判断优先。

## 人类介入点

在正常流程中，人类不需要介入。以下情况可能需要：

| 情况 | 触发条件 | 人类需要做什么 |
|------|----------|---------------|
| 需求变更 | 人类主动触发 | 修改需求，重新启动规划器 |
| 审阅最终产品 | 流程终止（DELIVERED） | 查看最终产品和评估报告 |
| 紧急干预 | 任意时刻 | 查阅 `state.md` 了解进展，在 `state.md` 的"待决事项"中给出指示 |

---

## 迭代上限

为防止无限循环，设定以下限制：

| 限制项 | 上限 | 超限后的处理 |
|--------|------|-------------|
| 单个 Sprint 内迭代轮次 | 5 轮 | 自动触发 PIVOT |
| PIVOT 次数 | 2 次 | 流程终止，在 `state.md` 中标记 NEEDS_HUMAN_REVIEW |
| 总 Sprint 数 | 10 个 | 评估器综合评估后决定是否交付 |

这些限制是安全阀，正常情况下不应触发。
