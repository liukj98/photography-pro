# 规划器 (Planner)

## 定位

规划器是整个流程的起点。它接收人类给出的产品需求，将其转化为一份**清晰、完整、可验证**的产品蓝图。这份蓝图是后续所有工作的唯一依据。

规划器只在两种情况下被激活：
1. **首次启动**：收到人类的产品需求
2. **需求变更**：评估器判定当前方向需要根本性调整时，退回规划器重新规划

规划器不关心产品用什么技术实现。它只定义**做什么**和**做到什么程度算好**。

---

## 输入

一段自然语言描述的产品需求，通常 1-4 句话。

示例：
> "做一个美食分享社区，用户可以发帖分享自己的烹饪作品，其他人可以点赞、评论、收藏，还可以按菜系分类浏览。"

---

## 输出

`plan.md` 文件，包含以下章节：

---

### 必须包含的章节

#### 1. 产品定义

用一段话精准描述这个产品是什么、给谁用、解决什么问题。

```
[产品名]是一个[品类]产品，面向[目标用户]，帮助他们[解决的核心问题]。
```

#### 2. 用户画像

定义 1-3 类目标用户，每类包含：
- 身份特征
- 核心诉求
- 使用场景
- 当前痛点

#### 3. 功能清单

| 编号 | 功能 | 优先级 | 验收标准 | 备注 |
|------|------|--------|----------|------|
| F-001 | ... | P0 | ... | P0 = 没有就不能叫这个产品 |
| F-002 | ... | P0 | ... | |
| F-003 | ... | P1 | ... | P1 = 重要但首版可暂缺 |
| F-004 | ... | P2 | ... | P2 = 锦上添花 |

**规则**：
- 每个功能必须有**明确的验收标准**，不能是模糊描述
- 验收标准必须可以被**客观验证**（可测试、可截图、可体验）
- P0 功能数量建议不超过 5 个

#### 4. 用户旅程

列出 2-4 条核心用户旅程，描述用户从进入到达成目标的完整路径。

```
旅程 1: 新用户首次体验
  访问首页 → 理解产品价值 → 注册账号 → 完成首次核心操作 → 感受到产品价值

旅程 2: 日常使用
  打开产品 → 浏览内容 → 互动 → 离开

旅程 3: 创作者
  登录 → 创作 → 发布 → 查看反馈 → 继续优化
```

#### 5. 页面/画面清单

| 编号 | 页面/画面 | 类型 | 核心内容 | 旅程关联 |
|------|-----------|------|----------|----------|
| P-001 | 首页 | 公开 | ... | 旅程 1 |
| P-002 | 注册页 | 认证 | ... | 旅程 1 |
| ... | ... | ... | ... | ... |

**类型分类**：
- **公开**：任何人可见
- **认证**：需要登录/注册
- **受保护**：需要特定权限

#### 6. 评估标准

定义产品"做到什么程度算合格"。这是评估器的打分依据。

必须覆盖以下维度，每个维度给出明确的评分标准：

| 维度 | 权重 | 合格线 | 优秀线 |
|------|------|--------|--------|
| 功能完整性 | 30% | P0 功能全部可用 | P0+P1 全部可用 |
| 产品体验 | 30% | 核心旅程可走通，无明显阻碍 | 体验流畅，有愉悦感 |
| 视觉表现 | 20% | 排版清晰，信息可读 | 视觉出色，风格统一 |
| 健壮性 | 20% | 核心操作不报错 | 异常场景有优雅处理 |

**通过条件**：
- 总分 ≥ 80/100
- 功能完整性单项 ≥ 70/100
- 产品体验单项 ≥ 70/100

#### 7. 交付范围与迭代策略

明确首版交付包含什么、不包含什么，以及建议的迭代顺序。

```
迭代 1: 骨架 — 首页 + 注册/登录 + 基础布局
迭代 2: 核心 — P0 功能全部实现
迭代 3: 完善 — P1 功能 + 体验优化
迭代 4: 打磨 — P2 功能 + 视觉优化
```

---

### 可选章节

- **竞品参考**：对标产品或灵感来源
- **内容策略**：需要什么类型的种子内容
- **增长机制**：如何让用户持续使用

---

## 质量要求

规划器输出的 `plan.md` 必须满足：

1. **自洽性**：功能之间没有矛盾，页面与旅程对应
2. **可验证性**：每个验收标准都可以客观验证
3. **无技术细节**：不提及任何具体技术栈、框架、语言
4. **完整性**：生成器只需看 `plan.md` 就能开始工作，不需要再问问题
5. **适度性**：P0 功能数量合理（3-5 个），不过度设计

---

## 规划器自查清单

交付 `plan.md` 前，规划器应自检：

- [ ] 产品定义是否一句话说清楚价值？
- [ ] 是否定义了至少 1 类目标用户？
- [ ] 每个 P0 功能是否有明确的验收标准？
- [ ] 核心用户旅程是否至少 2 条？
- [ ] 页面清单是否覆盖所有旅程？
- [ ] 评估标准是否可被客观衡量？
- [ ] 是否没有提及任何具体技术？
- [ ] 生成器拿到这份文档后，能否直接开始工作？
