# 总览：Harness 框架与 To C 项目结合

## 1. 什么是 Harness Engineering

Harness Engineering 是一套对抗性多智能体协作方法论，通过 **规范器、规划器、生成器、评估器、自动化测试** 五个角色的协作，实现高质量软件的自动化构建。

```
用户需求 → 规划器 → 生成器 → 自动化测试 → 评估器 → 反馈 → 迭代
```

## 2. 为什么 To C 项目需要 Harness

To C 产品有以下特点，天然适合 Harness 工程化：

| To C 特征 | Harness 对应能力 | 价值 |
|-----------|-----------------|------|
| 用户旅程复杂（注册→浏览→互动→付费） | E2E 测试覆盖用户旅程 | 保证核心链路不断裂 |
| 用户体验要求高 | 评估器设计质量维度 | 持续迭代 UI/UX |
| 功能迭代快 | Sprint 合同机制 | 控制每次交付质量 |
| 安全性要求高（用户数据） | 自动化诊断 Auth/RLS 错误 | 快速定位权限问题 |
| 需要快速验证想法 | 规划器快速产出 MVP 规划 | 缩短从需求到上线的时间 |

## 3. Harness 五大角色与 To C 的映射

### 3.1 规范器 (Spec) → To C 质量标准

定义 To C 产品的评估维度：

```
┌───────────────────────────────────────────────────┐
│                To C 评估维度                        │
├─────────────┬──────────────┬──────────────────────┤
│   维度       │   权重       │   To C 关注点         │
├─────────────┼──────────────┼──────────────────────┤
│ 功能验收     │   30%        │ 用户旅程完整性        │
│ 设计质量     │   25%        │ 视觉体验、响应式      │
│ 代码质量     │   20%        │ 类型安全、可维护性    │
│ 性能表现     │   15%        │ 首屏加载、交互流畅度  │
│ 安全性       │   10%        │ 用户数据保护、RLS     │
└─────────────┴──────────────┴──────────────────────┘
```

### 3.2 规划器 (Planner) → To C 产品规划

To C 产品规划的特别关注：

- **用户画像**：谁在用？为什么用？
- **核心用户旅程**：注册 → 首次体验 → 留存 → 付费/转化
- **增长机制**：分享、邀请、SEO、内容发现
- **冷启动方案**：测试数据填充、种子内容
- **MVP 范围**：最小可验证功能集

### 3.3 生成器 (Generator) → To C 代码实现

To C 项目的标准代码模式：

```
src/
├── components/
│   ├── ui/          # 通用 UI 组件（Button, Card, Toast...）
│   ├── layout/      # 布局组件（Header, Footer, Sidebar...）
│   ├── auth/        # 认证相关组件
│   └── interactions/ # 互动组件（点赞、评论、收藏...）
├── pages/           # 页面（每个页面一个文件）
├── hooks/           # 自定义 Hooks（useAuth, usePhotos...）
├── stores/          # Zustand 状态管理
├── types/           # TypeScript 类型定义
└── lib/             # 工具库、Supabase 客户端
```

### 3.4 评估器 (Evaluator) → To C 质量验收

To C 产品的关键验收清单：

- [ ] 用户注册/登录流程完整可用
- [ ] 页面在不同屏幕尺寸下正常显示
- [ ] 数据加载有 Loading 状态
- [ ] 错误操作有友好提示
- [ ] 核心用户旅程 E2E 测试通过
- [ ] 无 P0/P1 级别 Bug

### 3.5 自动化测试 (Automation) → To C 错误诊断

To C 项目常见错误模式：

| 错误 | 分类 | 诊断 | 自动修复 |
|------|------|------|----------|
| Email not confirmed | Auth | Supabase 邮箱验证开启 | 提示关闭 |
| RLS policy violation | Database | 缺少 INSERT/SELECT 策略 | 生成修复 SQL |
| Storage upload failed | Storage | Storage bucket 策略缺失 | 生成修复 SQL |
| Element not found | UI | 选择器错误/页面未加载 | 检查选择器 |
| 401 Unauthorized | Auth | Token 过期/权限不足 | 重新登录 |

## 4. 完整工作流（To C 场景）

```
1. 描述需求
   "做一个摄影社区，用户可以上传作品、浏览点赞、关注摄影师"
         ↓
2. 规划器生成 plan.md
   - 用户画像、功能列表、数据模型、页面清单
   - To C 特有：冷启动方案、增长机制
         ↓
3. Sprint 循环
   ┌─ 生成器实现功能
   │   - 搭建项目骨架
   │   - 实现用户认证（Supabase Auth）
   │   - 实现核心页面
   │   - 实现互动功能
   ├─ 自动化测试
   │   - auth.spec.ts（注册/登录流程）
   │   - explore.spec.ts（内容浏览）
   │   - interaction.spec.ts（点赞/收藏）
   │   - 错误诊断 + 自动修复
   └─ 评估器验收
       - 功能完整性
       - 设计质量
       - 性能指标
       - 安全检查
         ↓
4. 迭代直到达标（总分 ≥ 8.0）
         ↓
5. 交付上线
```

## 5. 与通用 Harness 的差异

| 维度 | 通用 Harness | To C Harness |
|------|-------------|--------------|
| 评估重点 | 功能正确性 | **用户体验 + 功能正确性** |
| 测试策略 | API/单元测试优先 | **E2E 用户旅程测试优先** |
| 错误诊断 | 通用错误模式 | **Auth/RLS/Storage To C 专有错误** |
| 规划模板 | 技术导向 | **用户导向（画像、旅程、增长）** |
| 质量阈值 | 总分 ≥ 8.0 | 总分 ≥ 8.0 + **核心旅程 100% 通过** |
| 数据管理 | 测试 Mock | **真实 Supabase + seed 脚本** |
